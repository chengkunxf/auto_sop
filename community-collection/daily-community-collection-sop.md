# 每日社群收集 SOP

## 目标

把 Reddit / Facebook / 其他社区的获客基础工作，做成一套 **每天可重复执行** 的流程。

这份 SOP 不是只管评论，也不是只管发帖，而是管整套上游内容更新：

1. 关键词池更新
2. 社群池更新
3. 内容草稿更新
4. 回复模板更新
5. 平台规则更新
6. 回复队列更新
7. 互动记录更新
8. 日报与总览更新

> 这份 SOP 的目标，是让 OpenClaw 每天都能把“基础盘”往前推，而不是只做零散动作。

## Step 1：更新关键词池（keywords）

### 目标
每天新增/优化一批关键词，让上游搜索源持续变厚，并且能直接服务于：

- 社群发现
- 内容草稿生成
- 评论回复
- 后续承接页文案

---

### 关键词收集的 5 个主要来源

#### 来源 1：网站实际产品页
先看站点当前真实在卖什么、怎么描述。

重点提取：
- 产品类型词
- 风格词
- 功能词
- 使用场景词

例如：
- crossbody bag
- shoulder bag
- camera bag
- lightweight bag
- minimalist bag
- casual bag
- everyday bag

> 原则：先从真实产品出发，不要一上来凭空扩词。

#### 来源 2：从现有产品词往外扩
不是只抄产品页原词，而是把原词向外扩一层。

例如：
- crossbody bag for women
- best everyday crossbody bag
- lightweight crossbody bag
- travel crossbody bag women
- small shoulder bag for daily use

优先扩这几类：
- 交易词
- 场景词
- 痛点词
- 比较词
- 社群搜索词

#### 来源 3：从社区 / 帖子标题里反推关键词
看目标 subreddit、Facebook 搜索结果、讨论标题里反复出现什么表达。

例如：
- everyday crossbody
- practical bag
- stylish but practical
- multiple compartments
- bag organization
- purse essentials
- shoulder vs crossbody

> 这些词通常更贴社区讨论语境，比纯电商词更适合用来找群和写内容。

#### 来源 4：从真实互动语言里收词
从 `engagements`、评论、回复里提用户原话。

例如：
- too heavy
- easy access
- bad pockets
- awkward strap
- cute but impractical
- black hole bag
- pouch system
- fits enough
- everyday essentials

> 这类词价值很高，因为它们是用户真实会说的话，后面可以同时反哺 keywords、content_drafts、reply_playbooks 和 landing page。

#### 来源 5：从社群搜索目标倒推关键词
把那些能直接用来找群/找 subreddit 的表达单独收进词池。

例如：
- handbag lovers
- women fashion accessories
- capsule wardrobe women
- travel accessories women
- crossbody bag women
- women bag recommendations
- cute and practical bags

> 这一类词的目标不是拿来卖货，而是直接拿去找社区。

### A. 关键词去重、分类与优先级判定

#### 1）先做去重，不要把词池越堆越脏
每天新加词前，先对当天候选词做一轮标准化。

**标准化规则：**
- 全部转成统一大小写格式（建议小写）
- 去掉明显无意义的标点差异
- 单复数、连字符、空格差异统一
- 明显同义表达合并到主词
- 品牌词 / 产品词 / 场景词不要混成一条

**去重判断规则：**
- 如果只是同一意思的轻微写法差异，只保留 1 个主词
- 如果是同一词的长尾扩展，则保留主词 + 高价值长尾词，不要全部删掉
- 如果词义相近但搜索场景不同，可以并存
- 如果一个词已经被更真实的用户表达替代，旧词可以降级，不一定立刻删除

**建议保留结构：**
- `primary_keyword`：主词
- `aliases`：别名 / 同义词 / 变体
- `duplicate_of`：如果是重复词，标记归属到哪个主词

#### 2）再做分类，不要只按感觉放
每个词至少要能回答两个问题：
- 它属于哪一类词
- 它主要拿来做什么

**category 分类原则：**
- `core-transaction`：直接贴产品或购买意图，例如 `crossbody bag`
- `scene`：使用场景，例如 `travel crossbody bag`
- `pain-point`：用户问题或不满，例如 `too heavy bag`
- `community-search`：用来找群找 subreddit 的词，例如 `bag recommendations`
- `user-language`：用户原话、社区常见表达，例如 `cute but impractical`

**intent 判断原则：**
- `transactional`：偏购买、对比、选品
- `research`：偏搜资料、找建议、做功课
- `discussion`：偏社区讨论、观点交流
- `community-search`：偏找群、找圈子、找讨论场

#### 3）最后做优先级，不是所有词都值得先用
优先级不要凭直觉，至少按下面 4 个维度判断：

1. **业务相关性**：和当前产品线是否直接相关
2. **社区可用性**：能不能直接拿去找社区、找帖子、找 thread
3. **内容可转化性**：能不能拿去写帖子、评论、承接页
4. **信号新鲜度**：是不是最近互动或最近搜索里新出现的表达

**建议分法：**
- `high`：同时满足强相关 + 能找社区/写内容 + 最近有真实信号
- `medium`：相关，但用途偏单一，或还需要验证
- `low`：太泛、过老、太弱相关，先保留观察

**高优先级词通常满足：**
- 贴实际产品
- 能用于找社区
- 能用于写内容
- 能用于做承接页文案
- 能反映真实用户痛点或真实语言

**低优先级词通常是：**
- 太泛
- 跟当前包类主线弱相关
- 当前承接不了的高交易词
- 对社群发现和内容生成帮助不大的词

### B. 关键词更新后的输出动作（反哺 `communities`）

关键词更新不能停在“词池变大”，而要立刻转成对 `communities` 的实际推进。

#### 1）先从当天高优先级词里挑 3–5 个输出词
只拿当天标记为 `high` 的词优先往下游传，不要全量无差别扩散。

优先选这几类：
- 能直接拿去搜 subreddit / Facebook Groups 的词
- 最近在互动里新出现、但之前没系统搜过的词
- 能明显打开新社区方向的词
- 能补足当前 communities 结构空白的词

#### 2）把关键词转成社区搜索 query
不要直接把原词生硬照搬，而是转成适合搜社区的查询表达。

**转法示例：**
- 产品词 → 社区兴趣词
  - `crossbody bag` → `crossbody bag lovers` / `crossbody bag women` / `bag recommendations`
- 场景词 → 人群 / 需求词
  - `travel crossbody bag` → `travel accessories women` / `travel bag recommendations`
- 痛点词 → 讨论词
  - `too heavy bag` → `lightweight bag` / `practical bag` / `bag organization`
- 用户语言词 → 社区自然表达
  - `cute but impractical` → `stylish but practical` / `everyday bag discussion`

#### 3）把 query 立刻反哺到 communities 的每日动作里
当天新出的高价值 query，要直接进入 `communities` 更新流程，而不是留到以后再说。

**至少做这 3 个动作：**
- 用新 query 搜 Reddit 社区
- 用新 query 搜 Facebook Groups
- 如果 query 更适合内容讨论，也同步标记给 `reply_queue` 作为 thread 搜索方向

#### 4）更新 `communities` 时要记录“这个社区是被哪个词带出来的”
每新增一个候选社区，至少要在现有字段范围内记录来源。

当前表里可直接承接的字段是：
- `source_keyword`
- `notes`（说明这个词为什么把它搜出来、是否值得继续追）

这样后面回看时，能知道：
- 哪些关键词真的能找到高质量社区
- 哪些词搜出来的都是噪音
- 哪些词值得反复复用

#### 5）做一次关键词 → 社区质量反馈
不是所有关键词都值得继续投喂给 `communities`，当天要做一轮轻反馈。

**建议按结果分 3 类：**
- `validated`：能稳定搜到相关、高活跃、可用社区
- `needs refinement`：方向对，但结果太泛、太弱，需要换写法
- `low yield`：搜出来基本没用，后续降级处理

这一步可以直接写进关键词的 `notes` 或附加状态字段里。

#### 6）如果关键词有效，要触发后续两个下游动作
当一个词被验证为 `validated`，不要只停在 communities。

要继续判断它是否还能反哺：
- `content_drafts`：这个词是否能生成新的讨论题 / 标题角度
- `reply_queue`：这个词是否能帮助找到新的 thread_type 或评论场景

也就是说，一个好关键词至少应该推动：
- 找到新社区
- 带来新内容方向
- 带来新回复机会

### Step 1 的最终产出要求
每天跑完关键词池更新后，至少要产出：
- 一批已去重、已分类、已标优先级的新关键词
- 当天最高优先级的 3–5 个词
- 由这些词转出来的社区搜索 query
- 至少一轮对 `communities` 的实际反哺动作
- 对有效 / 无效关键词的初步反馈记录

## Step 2：更新社群池（communities）

### 目标
持续扩充可用社区池，并维护优先级。

### 当前优先平台
1. Reddit
2. Facebook
3. Discord（补充）

### 每日建议量
- **50 条候选**（冷启动阶段）
  - Facebook：25–30
  - Reddit：15–20
  - 其他补充：0–10

### 找社区时先看什么
- 人群匹配度
- 话题相关度
- 活跃度
- 内容友好度
- 商业潜力
- 风险

### 社区不是“找到就算完成”，而是要先做初筛评分
选取社群 ≠ 社群已经选完。

更准确的流程是：
1. 先根据关键词和 query 找到候选社区
2. 对候选社区做一轮初筛评分
3. 按评分分成 `A / B / C`
4. 后续再根据真实观察结果复评

也就是说，`communities` 的作用不是只存名单，而是存：
- 候选来源
- 初筛判断
- 投入优先级
- 后续复评结果

### 初筛评分维度
建议先按下面 5 个核心维度评分：

1. **人群匹配度**
   - 这个社区的人，是否就是潜在目标人群
2. **活跃度**
   - 最近是否持续有新帖、新评论、新互动
3. **内容友好度**
   - 社区是否允许自然讨论、提问、经验分享，而不是极端排斥品牌或外部建议
4. **商业潜力**
   - 社区讨论是否和产品选择、推荐、对比、需求表达有关
5. **风险**
   - 是否容易删帖删评、规则严格、氛围敌对、强反营销

### 初筛评分方法（只依附现有字段）
先判断等级字段，再折算成分数写入 `overall_score`。

#### 1）等级字段怎么填
- `activity_level`：填 `high / medium / low`
- `content_friendliness`：填 `high / medium / low`
- `commercial_potential`：填 `high / medium / low`
- `risk_level`：填 `low / medium / high`

#### 2）各字段判断原则
**activity_level**
- `high`：近期持续有新帖、新评论、新互动
- `medium`：有更新，但频率一般
- `low`：更新慢、互动弱、明显不活跃

**content_friendliness**
- `high`：允许自然讨论、经验分享、提问，氛围开放
- `medium`：可以讨论，但对外部推荐、品牌相关内容较敏感
- `low`：明显排斥推广、限制很多、稍不自然就容易出问题

**commercial_potential**
- `high`：社区经常出现选购、推荐、比较、需求表达
- `medium`：有一定消费讨论，但不是主线
- `low`：主要是闲聊、展示或弱购买意图内容

**risk_level**
- `low`：规则可控，正常讨论风险低
- `medium`：规则较多，稍有不慎就可能踩线
- `high`：删帖删评风险高，社区对外部介入明显敏感

#### 3）overall_score 怎么算
为了不新增字段，`overall_score` 直接作为综合分。

**建议映射：**
- `high = 30`
- `medium = 20`
- `low = 10`

其中 `risk_level` 反向处理：
- `low risk = 10`
- `medium risk = 20`
- `high risk = 30`

**建议公式：**

`overall_score = audience_fit_score + activity_score + content_friendliness_score + commercial_potential_score - risk_score`

但因为表里没有单独的 score 字段，所以实际执行时按下面方式心算后写入 `overall_score`：
- `audience_guess` 对应的人群匹配度：高=30 / 中=20 / 低=10
- `activity_level`：高=30 / 中=20 / 低=10
- `content_friendliness`：高=30 / 中=20 / 低=10
- `commercial_potential`：高=30 / 中=20 / 低=10
- `risk_level`：低=10 / 中=20 / 高=30

例如：
- 人群匹配高（30）
- 活跃度高（30）
- 内容友好度中（20）
- 商业潜力高（30）
- 风险低（10）

则：
`overall_score = 30 + 30 + 20 + 30 - 10 = 100`

### `tier` 的 A / B / C 分层建议
#### 分数阈值
- `A`：`overall_score >= 80`
- `B`：`overall_score = 60–79`
- `C`：`overall_score < 60`

#### A 类
建议满足：
- `overall_score >= 80`
- `risk_level` 不高
- 人群匹配明确
- 值得优先投入

**动作：**
- 优先加入 shortlist
- 优先用于找 thread
- 优先进入 `reply_queue` 上游
- 后续优先做复评

#### B 类
建议满足：
- `overall_score` 在 `60–79`
- 有一定价值，但还要继续观察
- 暂时不作为最高优先级

**动作：**
- 留在池子里
- 后续按时间补观察
- 不作为最优先投放目标

#### C 类
建议满足：
- `overall_score < 60`，或
- `risk_level` 偏高，或
- 人群与主题弱相关

**动作：**
- 保留记录即可
- 暂不进入重点动作
- 只有在关键词方向变化时再回看

### 什么时候做复评
以下情况要更新评分和 tier：
- 实际看过社区帖子和评论后
- 发现规则比预期更严或更松
- 社区活跃度变化明显
- 社区对推荐、讨论、评论的接受度和预期不一致
- 已经从该社区拿到了真实互动结果

### Step 2 的核心产出
每天跑完 `communities` 后，至少要有：
- 一批新增候选社区
- 基于现有表结构记录的基础信息
- 每个候选社区的初筛评分
- 清晰的 `A / B / C` 分层
- 一份当天优先值得继续投入的 shortlist

## Step 3：更新内容草稿（content_drafts）

### 目标
持续为高优先社区补充可发内容。

### 输入
- `communities`
- `keywords`
- `engagements`

### 优先生成的内容类型
- discussion
- comparison
- feedback_request
- tips

### 每日建议量
- 新增或优化 **3–8 条**

### 生成逻辑
优先围绕：
- 最近互动里出现的真实痛点
- 最近高频关键词
- A/B 类社区最容易接的话题

### 写入字段（以实际 Excel 为准）
- date
- community_name
- platform
- content_angle
- draft_title
- draft_body
- cta_type
- risk_note
- review_status
- publish_status
- target_url
- post_link
- notes

### 现有字段怎么用
- `content_angle`：填内容类型或切入角度，例如 `discussion / comparison / feedback_request / tips`
- `draft_title`：填标题或主问题句
- `draft_body`：填正文草稿
- `cta_type`：填预期互动方式，例如提问、征求经验、求建议
- `risk_note`：填这条草稿的风险提醒，例如是否容易像推广、是否可能踩版规
- `review_status`：标是否待审核、可直接用、需重写
- `publish_status`：标未发、待发、已发
- `target_url`：如果这条内容有明确承接页，再填；没有就留空
- `post_link`：发出后再回填
- `notes`：记录为什么写这条、适合哪个社区、后续怎么改

### 产出原则
- 冷启动阶段优先弱商业、强讨论
- 不急着强导流
- 先拿真实回复和真实语言
- 每条草稿都要能说明“适合哪个社区、想引出什么讨论、风险在哪里”

### Step 3 的核心产出
每天跑完 `content_drafts` 后，至少要有：
- 一批基于高优先社区生成的草稿
- 每条草稿明确的 `content_angle`
- 已填写的 `review_status / publish_status`
- 对风险和适用社区的简短说明

## Step 4：更新回复模板（reply_playbooks）

### 目标
让评论系统持续长脑子，不是每次临场乱写。

### 输入
- `engagements`
- `reply_queue`
- 新发现的 `thread_type`

### 每日更新内容
- 补新 subreddit 的模板
- 补新 `thread_type` 的模板
- 淘汰低质量模板
- 根据真实反馈微调语气

### 写入字段（以实际 Excel 为准）
- platform
- community_name
- thread_type
- reply_goal
- trigger_pattern
- tone
- template_text
- risk_level
- status
- notes
- last_updated

### 现有字段怎么用
- `thread_type`：定义模板适配的讨论场景，例如 `weight_pain / comparison / recommendation_request`
- `reply_goal`：写模板的目的，例如共情、提供经验、追问细节、自然参与讨论
- `trigger_pattern`：写什么情况下触发这条模板，例如楼主提到重量、收纳、肩带不舒服、通勤使用
- `tone`：写语气风格，例如 practical / empathetic / casual / neutral
- `template_text`：写可复用的基础回复，不要求一字不改照抄，但要能直接改成候选回复
- `risk_level`：建议填 `low / medium / high`
- `status`：建议至少区分 `approved / testing / paused`
- `notes`：补充适用边界、禁用场景、使用提醒
- `last_updated`：记录最后一次调整时间

### 模板更新原则
优先保留的模板：
- 能贴合真实 `thread_type`
- 语气自然，不像推广
- 能引出继续交流
- 已经在真实互动里证明可用

优先淘汰或重写的模板：
- 太像统一话术
- 太长
- 太像品牌引导
- 在多个社区里连续表现差

### 每日建议量
- 优化 **2–5 条模板**
- 有新社区时再新增模板

### Step 4 的核心产出
每天跑完 `reply_playbooks` 后，至少要有：
- 新增或优化后的模板
- 每条模板明确的 `thread_type / reply_goal / trigger_pattern`
- 清楚的 `risk_level / status`
- 对适用边界和语气的简短说明

## Step 5：更新平台规则（posting_rules）

### 目标
把平台/社区的动作边界持续结构化。

### 什么时候更新
- 遇到删帖/删评
- 发现新规则
- 社区氛围变化
- 某类动作风险明显变化

### 写入字段（以实际 Excel 为准）
- platform
- community_name
- action_type
- allowed
- priority
- rule_summary
- do_not_do
- notes
- last_checked

### 现有字段怎么用
- `action_type`：写动作类型，例如 `comment / post / link / followup`
- `allowed`：写是否允许，例如 `yes / no / caution`
- `priority`：写规则重要程度，建议 `high / medium / low`
- `rule_summary`：简写该社区当前可做什么
- `do_not_do`：直接写不能做什么，越具体越好
- `notes`：补充实操观察，例如当前账号阶段、版主风格、删帖特征、频率建议
- `last_checked`：最后一次确认时间

### 规则更新原则
优先把这些信息写清楚：
- 评论是否允许
- 主帖是否允许
- 是否能提产品、链接、店铺
- 哪类 thread 可以碰，哪类 thread 不碰
- 当前阶段更适合低频评论还是可以做讨论帖

### 每日动作
- 更新被删原因
- 更新风控观察
- 更新日量限制建议
- 更新哪些 thread 不碰
- 把新发现的规则变化同步给 `reply_queue` 和 `content_drafts`

### Step 5 的核心产出
每天跑完 `posting_rules` 后，至少要有：
- 关键社区当前的动作边界
- 每类动作对应的 `allowed / priority`
- 清楚的 `rule_summary / do_not_do`
- 能直接供 `reply_queue` 和 `content_drafts` 使用的风险判断

## Step 6：更新回复队列（reply_queue）

### 目标
把“今天可以回什么”变成明确任务队列。

### 输入
- `communities`
- `posting_rules`
- `reply_playbooks`

### 每日动作
1. 找新线程
2. 标 `thread_type`
3. 匹配 `playbook_ref`
4. 生成 `candidate_reply`
5. 标 `priority / risk_level / queue_status`

### 每日建议量
- 新增 **8–12 条** 候选
- 其中筛出 **1–3 条 approved**

### 写入字段（以实际 Excel 为准）
- date
- platform
- community_name
- thread_url
- thread_title
- thread_type
- target_comment_summary
- reply_goal
- playbook_ref
- candidate_reply
- priority
- risk_level
- queue_status
- assigned_to
- scheduled_at
- posted_at
- result_status
- engagement_signal
- notes

### 现有字段怎么用
- `thread_type`：标这个帖子属于哪类场景，例如求推荐、求比较、吐槽痛点、经验讨论
- `target_comment_summary`：简写楼主问题或目标评论上下文
- `reply_goal`：写这条回复想达到什么，例如参与讨论、提供经验、试探兴趣、引导后续交流
- `playbook_ref`：引用对应模板或模板编号
- `candidate_reply`：填候选回复正文
- `priority`：建议填 `high / medium / low`
- `risk_level`：建议填 `low / medium / high`
- `queue_status`：建议至少区分 `draft / approved / posted / skipped`
- `assigned_to`：谁来执行
- `scheduled_at`：准备什么时候发
- `posted_at`：实际发出时间
- `result_status`：发出后记录结果，例如已发、被删、无回复、有互动
- `engagement_signal`：记录互动强弱，例如 low / medium / high 或简短描述
- `notes`：补充为什么值得回、为什么暂缓、要注意什么

### 入队判断原则
优先进入 `reply_queue` 的线程通常满足：
- 来自 `A` 类社区，或高质量 `B` 类社区
- 主题与当前关键词、产品场景、用户痛点相关
- 有明确可回答的问题或讨论空间
- 按 `posting_rules` 判断风险可控

不优先入队的情况：
- 明显高风险或容易被判推广
- 主题过弱相关
- 没有自然回复空间
- 与已有候选重复度太高

### Step 6 的核心产出
每天跑完 `reply_queue` 后，至少要有：
- 一批新候选线程
- 每条候选对应的 `thread_type / playbook_ref / candidate_reply`
- 已填写的 `priority / risk_level / queue_status`
- 至少 `1–3` 条可执行的 `approved` 项

### 当前推荐执行节奏
- 评论优先
- 每天低频
- 先 approved 低风险项

# 五、每日最小产出门槛与 fallback 规则

## 最小产出门槛
每天跑完后，至少满足：
- `keywords >= 10`
- `reply_queue >= 20`

如果未达到，就不能视为当天流程已完成，必须继续补量。

## fallback 规则 1：如果 `keywords < 10`
按下面顺序继续补：
1. 回到站内产品源
   - 首页主推区
   - collection / category 页
   - product detail 页
2. 从 `engagements` 提取新的真实用户语言
3. 从 `communities` 和相关 thread 标题补高频表达
4. 对当天最有价值的词再做一轮扩展

### `keywords < 10` 时的执行原则
- 优先补高相关词，不为了凑数加泛词
- 优先补能直接反哺 `communities` 和 `reply_queue` 的词
- 如果补到 10 条后仍然质量很弱，继续补到有可用词为止，而不是机械停下

## fallback 规则 2：如果 `reply_queue < 20`
按下面顺序继续补：
1. 先深挖现有 `A / B` 类社区里的新 thread
2. 优先找最近、活跃、可自然回复的 thread
3. 如果现有社区不够，再回到 `communities` 扩新候选
4. 如果新社区还是不够，再回到 Step 1 补新关键词，再重新搜 thread

### `reply_queue < 20` 时的执行原则
- 优先补低风险、可执行的真实线程
- 先补现有高质量社区，再扩新社区
- 不要为了凑数把明显高风险、弱相关、无自然回复空间的 thread 塞进队列
- 如果队列达到 20，但大多数都是低质量候选，也不算真正达标

## completion 判断
只有当下面两个条件同时满足，才算当天这一轮基础流程完成：
- `keywords >= 10`
- `reply_queue >= 20`
